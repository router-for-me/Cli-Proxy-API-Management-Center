## 1 技术栈约束

### 1.1 核心栈

- React 19 + TypeScript 5.9 + Vite 7
- 样式：SCSS Modules（`*.module.scss`）
- 状态管理：zustand 5
- 国际化：i18next（en/zh-CN/zh-TW/ru 四语）
- 包管理器：**bun**（强制，不使用 npm）

### 1.2 禁止事项

- 不引入 `package.json` 未声明的新依赖；如需新依赖，先与用户确认
- 不使用 npm，不使用 pnpm（特殊场景除外，需确认）
- 不引用 csdn.net、阿里云/腾讯云/华为云社区等内容农场信息
- 新增 UI 文案必须同时在 `src/i18n/zh`、`src/i18n/en`、`src/i18n/zh-TW`、`src/i18n/ru` 补键

### 1.3 CLI 工具偏好

- 优先使用 Rust 现代 CLI（ripgrep 替代 grep、fd 替代 find）

## 2 代码执行原则

### 2.1 约束四原则

1. **处理不确定性与信息缺失**：做出假设前必须提出澄清问题，澄清问题出现在动手编码之前
2. **管理复杂性**：保持代码和 API 简洁，优先选择最简单的实现方式，避免过度设计
3. **维护代码库连贯性**：不得移除或更改不完全理解的代码或注释；禁止"顺带优化"的冗余提交
4. **设定清晰成功标准**：多步骤任务每一步都有明确可检查的完成标准；diff 只出现被要求修改的部分

### 2.2 编码执行流

编码前三思 → 必读现有文件（已读勿重读） → 优先局部修改、禁全盘重写 → 推理全面透彻、输出简洁明了

### 2.3 脚本执行安全

- 禁止无备份的全量复写/删除/覆盖
- 破坏性动作（`rm -rf`、`git reset --hard`、`git push --force`）必须先 echo、询问 Y/n（默认 n）、记录日志
- 脚本失败时返回非零退出码并打印清晰的中文错误原因

## 3 目录与文件约定

### 3.1 源码目录

```
src/
├── App.tsx              # 根组件
├── main.tsx             # 入口
├── components/          # 通用组件（common/config/layout/ui/monitor/usage/...）
├── features/            # 功能模块
├── hooks/               # 自定义 Hook（12 个）
├── i18n/                # 国际化（en/zh-CN/zh-TW/ru）
├── pages/               # 页面组件（按功能命名）
├── router/              # 路由配置
├── services/api/        # API 客户端层
├── services/storage/    # 本地存储封装
├── stores/              # zustand 状态仓库
├── styles/              # 全局样式
├── types/               # TypeScript 类型定义
└── utils/               # 工具函数与常量
```

### 3.2 文件命名规范

- 页面组件：`{功能}Page.tsx` + `{功能}Page.module.scss`
- Hook：`use{功能}.ts`
- API 客户端：`{领域}.ts`（如 `usageSqlite.ts`、`authFiles.ts`）
- 类型定义：按领域组织在 `types/` 下
- SCSS Modules：与组件同名 `.module.scss`

### 3.3 本地存储键名规范

统一前缀 `cli-proxy-`，如 `cli-proxy-auth`、`cli-proxy-theme`、`cli-proxy-language`

## 4 提交规范

### 4.1 Conventional Commits（双语）

格式：`<type>(<scope>): <中文简述> [en: <english brief>]`

- `type`：feat | fix | refactor | perf | docs | chore | ci | test | build | style | revert | merge
- feat/fix/refactor/perf/build/ci/test/revert/merge **必须双语**
- docs/chore/style 可仅中文

### 4.2 分支规则

- `main`：稳定主干，禁止直接 commit
- `dev`：日常研发分支
- `feature/<topic>` / `fix/<topic>` / `refactor/<topic>`：从 dev 切出，merge 回 dev
- 禁止从 main 切功能分支

### 4.3 自动化流程

```
bun run feature <topic>   → 切功能分支
bun run commit            → 双语提交（可反复）
bun run promote           → 预检 + 合回 dev + 可选合入 main
bun run release           → 打 tag 触发 CI（仅 main）
```

### 4.4 AI 操作限制

- 禁止擅自 `git push` 或 `gh pr create`；必须先等用户 Y/N 确认
- 非交互环境可使用 `ALLOW_MAIN_PUSH=YES git push origin main`

## 5 调试判据

合并到 `main` 前必须全部 PASS：

| 检查项 | 命令 | 判据 |
| --- | --- | --- |
| TypeScript | `bun run type-check` | 无错误 |
| Lint | `bun run lint` | 无新增错误（上游既有问题可接受） |
| 构建 | `bun run build` | 成功 |
| 浏览器冒烟 | 手动验证 | Dashboard / Usage / Config / 监控中心无报错 |

一键检查命令：`bun run check`（type-check → lint → build）

## 6 Git 钩子

| 钩子 | 行为 |
| --- | --- |
| `pre-commit` | 禁止在 main 分支直接 commit；运行 `bun run type-check` |
| `commit-msg` | 校验 Conventional Commits 格式；feat/fix 等必须双语 |
| `pre-push` | 向 origin/main 推送时要求 YES 确认 |

## 7 计划与输出规范

### 7.1 计划前 6 步

1. 思维链：信息 → 分析 → 思考 → 质疑 → 评估 → 循环
2. 一句话重述目标（用户价值 + 验收标准）
3. 信息不足时提最少 3 个澄清问题
4. 对比「提出的路径」和「最短可行路径」
5. 只做与当前目标直接相关的最小改动
6. 补充建设性建议，必须交叉验证

### 7.2 最小输出模板

改动清单（文件 + 目的） → 风险点 → 建设性建议 → 交叉/边界/对抗验证
