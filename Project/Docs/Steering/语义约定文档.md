# 语义约定文档

> 本文档定义 CPA Dashboard 项目的核心术语和业务概念，确保 AI 与开发者对领域模型的理解一致。

---

## 1 项目身份

### CPA (CLI Proxy API)

CLI Proxy API 的缩写。本项目的管理面板（Dashboard）用于管理和监控 CLIProxyAPI 代理服务。在代码、文档、API 路径中统一使用 `cpa` 或 `CLIProxyAPI`。

### CPA Dashboard

本项目本身的名称。Web 管理面板，提供对 CLIProxyAPI 代理服务的可视化管理界面。仓库名 `CPA-Dashboard-kelen`。

### CLIProxyAPI

被管理的目标后端代理服务。本项目不直接修改 CLIProxyAPI 源码，仅通过管理 API 进行交互。后端改动通过提案文档（`.ai-local/Docs/proposals/`）提交给目标方。

---

## 2 核心业务概念

### 认证文件 (Auth Files)

CLIProxyAPI 用于身份验证的文件（如 API 密钥文件）。管理面板提供上传、编辑、删除等操作。对应 API 端点 `/v0/management/auth-files`。

### OAuth 卡片

管理面板中展示各 OAuth 提供商（Codex / Anthropic / Gemini CLI / Kimi 等）集成状态的卡片组件。每个卡片对应一个 OAuth 提供商的配置入口。

### 模型别名 (Model Alias)

为 AI 模型设置的别名映射，用于在 CLIProxyAPI 中简化模型引用。对应 `AuthFilesOAuthModelAliasEditPage`。

### 使用量 (Usage)

CLIProxyAPI 代理服务的请求使用统计数据，包含请求数、Token 消耗、延迟等指标。

## 3 SQLite 适配层

### 双通道 (Dual Channel)

使用量数据的两种获取路径：
- **原始通道**：CLIProxyAPI 内存队列，重启即丢失
- **SQLite 通道**：本地 SQLite 持久化存储，支持历史查询

### 双通道自动切换

前端 MonitorPage 组件的智能降级机制：优先尝试 SQLite 通道获取数据，若 SQLite 未启用或请求失败，自动降级到原始通道。对应 `useSqliteUsage` Hook 中的 `sqliteEnabled` 状态。

### 零侵入原则 (Zero-Intrusion)

SQLite 持久化方案的核心架构原则：
- 不修改 CLIProxyAPI 现有请求处理路径或业务逻辑
- 通过 `usage.Plugin` 接口自注册，仅在 `config.yaml` 显式启用时生效
- 使用 Go `init()` 函数自动注册，不改动原有代码流程

### sqliteAdapter

数据格式转换适配层。将 SQLite 的扁平 `UsageRecord[]` 格式转换为 MonitorPage 子组件期望的嵌套 `UsageData` 格式（按 api_key → model → details 层级组织）。

### usageSqliteApi

SQLite 持久化使用量的 API 客户端模块。对接 `/v0/management/usage-sqlite/*` 端点，提供统计、记录查询、模型统计、日统计等接口。

### useSqliteUsage

React Hook，封装 SQLite 使用量数据的获取、轮询、降级逻辑。返回值包含 stats、records、modelStats、dailyStats、sqliteEnabled 等状态。

## 4 API 路径约定

### 管理 API 前缀

所有管理 API 使用统一前缀 `/v0/management/`（常量 `MANAGEMENT_API_PREFIX`）。

### 端点命名规范

| 端点路径 | 说明 | 数据源 |
| --- | --- | --- |
| `/usage` | 原始使用量 | 内存队列 |
| `/usage-sqlite/stats` | 聚合统计 | SQLite |
| `/usage-sqlite/records` | 分页记录查询 | SQLite |
| `/usage-sqlite/model-stats` | 按模型统计 | SQLite |
| `/usage-sqlite/daily-stats` | 按日统计 | SQLite |
| `/usage-sqlite/enabled` | SQLite 是否启用 | 运行时检测 |

### 超时分级

- `TIMEOUT_DEFAULT`（30s）：常规请求
- `TIMEOUT_HEAVY`（60s）：大数据量查询（如 SQLite 使用量查询）

---

## 5 时间与查询约定

### Go Duration 格式

SQLite 使用量查询的 `since` 参数使用 Go duration 格式：
- `24h` — 最近 24 小时
- `7d` — 最近 7 天
- `720h` — 最近 30 天（默认值，常量 `SQLITE_USAGE_DEFAULT_SINCE`）

### 默认查询参数

| 参数 | 默认值 | 常量名 |
| --- | --- | --- |
| since | `720h`（30 天） | `SQLITE_USAGE_DEFAULT_SINCE` |
| limit | 100 | `SQLITE_USAGE_DEFAULT_LIMIT` |
| max limit | 10,000 | `SQLITE_USAGE_MAX_LIMIT` |
| refresh interval | 30,000ms | `SQLITE_USAGE_REFRESH_MS` |

## 6 提案机制

### 提案文档 (Proposal)

当需要对 CLIProxyAPI 后端做改动时，不直接修改目标项目源码，而是通过提案文档（位于 `.ai-local/Docs/proposals/`）提供完整的技术方案和改动清单，由目标方评估与实施。

### 前端就绪 / 待目标方实施

描述 SQLite 适配层当前状态的术语：前端 API 客户端、Hook、适配器已开发完成并接入页面，但后端 SQLite 持久化功能尚未在 CLIProxyAPI 中实现，需等待目标方根据提案实施。
